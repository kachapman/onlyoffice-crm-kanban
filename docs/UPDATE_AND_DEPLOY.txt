================================================================================
  Vanguard CRM Dashboard — Update & deploy workflow
  Repo: https://github.com/kachapman/onlyoffice-crm-kanban
  Production: https://dashboard.vanguardadj.com
================================================================================

Use this checklist every time you change the app on your desktop and publish
to the server.


--------------------------------------------------------------------------------
PART A — ON YOUR DESKTOP (development machine)
--------------------------------------------------------------------------------

1. Stop the local dev server (if it is running)

   cd ~/crm-kanban
   fuser -k 8765/tcp 2>/dev/null || true

   (Or press Ctrl+C in the terminal that ran ./start.sh)

2. Edit code

   Main files:
   - public/app.js, public/styles.css, public/index.html  (UI)
   - server.py, user_profile_store.py, notes_store.py, ics_calendar.py, crm_bot_store.py  (backend)
   - telegram_bot.py  (Telegram bot process — deploy separately)
   - docker-compose.yml, Dockerfile, deploy/*  (production)

3. Test locally (optional but recommended)

   cd ~/crm-kanban
   cp config.example.env .env    # only first time; then edit .env
   ./start.sh

   Open: http://127.0.0.1:8765
   Log in with your OnlyOffice CRM credentials.

   When done testing, stop the server again (step 1).

4. Commit and push to GitHub

   cd ~/crm-kanban
   git status
   git add <files you changed>
   git commit -m "Short description of the change"
   git push origin main

   Notes:
   - Do NOT commit .env (secrets) or data/ (user profiles).
   - SSH push: git@github.com:kachapman/onlyoffice-crm-kanban.git
   - HTTPS push: https://github.com/kachapman/onlyoffice-crm-kanban.git


--------------------------------------------------------------------------------
PART B — ON THE SERVER (DigitalOcean droplet)
--------------------------------------------------------------------------------

  IMPORTANT: Do one step, run its VERIFY block, then continue only if it passes.
  Full v1.1 walkthrough: docs/DEPLOY_v1.1_VERIFY_STEPS.md

  This update does NOT change DNS, estimate-nginx config, or TLS.

5. SSH in

   ssh root@159.89.229.126

   VERIFY: hostname shows the droplet; you are logged in.

6. (Optional) DNS check — read only

   dig +short dashboard.vanguardadj.com A @8.8.8.8

   VERIFY: 159.89.229.126 — if wrong, fix Bluehost before deploying.

7. Backup user profiles (recommended)

   cd /opt/vanguard/onlyoffice-crm-kanban
   docker run --rm -v onlyoffice-crm-kanban_dashboard-data:/data -v $(pwd):/backup alpine \
     tar czf /backup/dashboard-data-$(date +%F).tar.gz -C /data .

   VERIFY: ls -lh /backup/dashboard-data-*.tar.gz shows a new non-zero file.

8. Pull v1.4

   cd /opt/vanguard/onlyoffice-crm-kanban
   git fetch origin
   git checkout main
   git pull --ff-only origin main
   # Or: git checkout v1.4.0

   VERIFY: git log -1 --oneline; cat VERSION → 1.4.0; git status clean.

9. Update .env with any new environment variables (adds to existing)

   Only needed when new vars were added to config.example.env since last deploy.
   Do this BEFORE the docker compose build — the container reads .env at startup.

   nano /opt/vanguard/onlyoffice-crm-kanban/.env

   Document Server (OnlyOffice) — REQUIRED for document editing to work:

   DOCS_JWT_SECRET=change_me_in_production
   DOCS_PUBLIC_URL=https://docs.publicadjustermidwest.com
   # When Document Server is on a separate droplet, set DOCS_INTERNAL_URL to the
   # same public URL. When co-located in Docker, set the container hostname/port
   # (e.g. https://onlyoffice-docserver:443). Never use the legacy placeholder
   # http://docserver:8080 in production.
   DOCS_INTERNAL_URL=https://docs.publicadjustermidwest.com
   CRM_PUBLIC_URL=https://dashboard.publicadjustermidwest.com
   DOCSERVER_CONTAINER_NAME=onlyoffice-docserver

   VERIFY: grep -E "DOCS_JWT_SECRET|DOCS_PUBLIC_URL|DOCS_INTERNAL_URL|CRM_PUBLIC_URL|DOCSERVER_CONTAINER_NAME" .env

   If deploying the Telegram bot for the first time, ensure these lines exist
   (ask the developer for the actual values):

   TELEGRAM_BOT_TOKEN=your_token_here
   BOT_CRM_EMAIL=bot@vanguardadj.com
   BOT_CRM_PASSWORD=your_password_here

   VERIFY: grep -E "TELEGRAM_BOT_TOKEN|BOT_CRM_EMAIL|BOT_CRM_PASSWORD" .env

10. Rebuild and restart (CRM container only)

   docker compose build
   docker compose up -d
   # Or: ./deploy/update.sh

   VERIFY: docker compose ps → dashboard "running".

   IMPORTANT (post-v1.3+): Ensure Dockerfile COPY line includes ALL .py files imported by server.py
   (especially presence_store.py for the Team/Presence feature). Missing modules cause
   immediate crash on "python3 server.py" → container Restarting(1) → nginx 502.
   See the v1.4.5 release notes and the Dockerfile in the repo for the correct list.

11. Local app health (no nginx)

   curl -s http://127.0.0.1:8765/api/config

   VERIFY: JSON with portalUrl. If refused, docker compose logs dashboard.

12. Docker network (nginx → dashboard)

   docker exec estimate-nginx wget -qO- http://vanguard-crm-dashboard:8765/api/config

   VERIFY: same JSON. If fail:
   docker network connect estimate-enhancer_estimate-network vanguard-crm-dashboard
   (retry wget before editing nginx.conf)

13. Public site

   curl -sI https://dashboard.vanguardadj.com/ | head -5
   curl -s https://dashboard.vanguardadj.com/api/config

   VERIFY: HTTP 200; JSON from /api/config. If 502 only here, nginx -t && reload
   estimate-nginx (see PRODUCTION_SERVER_NOTES.txt).

14. Browser

   https://dashboard.vanguardadj.com — login, notifications feed, notes File menu.

   VERIFY: data and tiles unchanged; v1.1 UI (feed loading, notes archive menu).

15. (If deploying the Telegram bot for the first time) Set up venv & systemd

   a) Create the virtualenv (one time):

      python3 -m venv /tmp/botenv
      /tmp/botenv/bin/pip install python-telegram-bot==22.8 httpx

   b) Copy & enable the systemd service (one time):

      cp docs/crm-telegram-bot.service /etc/systemd/system/
      systemctl daemon-reload
      systemctl enable --now crm-telegram-bot

   VERIFY: systemctl status crm-telegram-bot → "active (running)"
          journalctl -u crm-telegram-bot -n 5 --no-pager | grep "200 OK"
          Message @vanguardupdates_bot on Telegram → expect a response.

   NOTE: The bot's required env vars (TELEGRAM_BOT_TOKEN, BOT_CRM_EMAIL,
   BOT_CRM_PASSWORD) must be in /opt/vanguard/onlyoffice-crm-kanban/.env
   BEFORE the docker compose rebuild (step 10), because the dashboard
   container reads .env at startup and uses TELEGRAM_BOT_TOKEN to
   authenticate bot API requests. If you forgot, run "docker compose up -d"
   (without --build) after adding them — Docker re-reads .env on every start.

16. (After code changes that touch the bot) Restart the systemd service

   systemctl restart crm-telegram-bot

   VERIFY: journalctl -u crm-telegram-bot -n 5 --no-pager → no "409 Conflict"


--------------------------------------------------------------------------------
NGINX / TLS (only when you change hostnames or certs)
--------------------------------------------------------------------------------

As of 2026-07, public traffic for dashboard.publicadjustermidwest.com is handled
by the HOST's nginx (systemd), not the Docker estimate-nginx container.

   Active site file on server:

     /etc/nginx/sites-enabled/dashboard.publicadjustermidwest.com

   Required inside the https server block for uploads to work:

     client_max_body_size 100m;
     proxy_request_buffering off;
     proxy_read_timeout 120s;

   Reload after editing the host site:

     sudo nginx -t
     sudo systemctl reload nginx

   Renew certificate (if needed):

     certbot certonly --webroot -w /var/www/certbot -d dashboard.publicadjustermidwest.com


--------------------------------------------------------------------------------
DOCKER NETWORK (if dashboard cannot be reached after recreate)
--------------------------------------------------------------------------------

The container binds to 127.0.0.1:8765. Host nginx proxies to that address.
The external network declaration is kept for compatibility but is not required
for public routing.

   Manual reachability test from host:

     curl -s http://127.0.0.1:8765/api/config

   If other containers on the old network need to reach it:

     docker network connect estimate-enhancer_estimate-network vanguard-crm-dashboard 2>/dev/null || true


--------------------------------------------------------------------------------
TROUBLESHOOTING
--------------------------------------------------------------------------------

| Problem                         | What to check                                      |
|---------------------------------|----------------------------------------------------|
| git pull fails                  | HTTPS clone or add SSH key on server               |
| 502 / empty site                | docker compose logs dashboard                      |
| Login OK locally, not production| HTTPS required (COOKIE_SECURE=true)                |
| Settings not saved              | docker volume dashboard-data; logs dashboard       |
| Old code after pull             | docker compose build --no-cache && up -d           |
| Bot 409 Conflict                | Only one bot instance per token allowed. Kill competing processes (`pkill -f telegram_bot`), then `systemctl restart crm-telegram-bot` |
| Bot fails to start              | Check .env has TELEGRAM_BOT_TOKEN, BOT_CRM_EMAIL, BOT_CRM_PASSWORD. Run manually first: `/tmp/botenv/bin/python3 telegram_bot.py` |
| Bot 503 "TELEGRAM_BOT_TOKEN not configured" | Dashboard container doesn't have the env var. Add to .env, then `docker compose up -d` (re-reads .env). No rebuild needed. |
| Dockerfile missing .py          | If server.py imports a new module, add it to the Dockerfile COPY line. Missing modules cause container crash → nginx 502. |
| Document editor won't open / "auth missing document.key" | OnlyOffice 7.1+ requires a plain-string document.key. Ensure code is current (v1.95.17+) and DOCS_JWT_SECRET matches the Document Server's JWT_SECRET. |
| "Document Server connection lost" | Check DOCS_INTERNAL_URL and CRM_PUBLIC_URL in .env. CRM_PUBLIC_URL must be reachable from the Document Server for downloads/callbacks. |
| Restart Document Server button fails | If Document Server runs on a separate droplet, the container won't exist on the dashboard host; restart it on its own host instead. |

VERSIONING & ROLLBACK
--------------------------------------------------------------------------------

Yes — the project is versioned with Git. Every commit on GitHub is a restore
point for CODE. There is no separate "app version number" in the UI; use Git
commits or tags.

What is versioned:
  - All source (public/, server.py, Docker files, deploy/) in GitHub
  - Branch: main (default)

What is NOT rolled back by git checkout:
  - Server .env (secrets; stays on droplet only)
  - User dashboard settings in Docker volume dashboard-data (profiles, notes)
  - /opt/estimate-enhancer/nginx.conf (back up manually before nginx edits)
  - Let's Encrypt certificates

Mark this working production baseline (do once, or after each stable release):

  cd ~/crm-kanban
  git tag -a v1.0.0-production-2026-06-05 -m "Working: dashboard.vanguardadj.com deployed"
  git push origin v1.0.0-production-2026-06-05

List tags:  git tag -l
List history: git log --oneline -15

ROLLBACK — code on the server (to a previous commit or tag)

  ssh root@YOUR_DROPLET_IP
  cd /opt/vanguard/onlyoffice-crm-kanban

  # See history
  git fetch origin
  git log --oneline -15
  git tag -l

  # Option A: roll back to a tag
  git checkout v1.0.0-production-2026-06-05

  # Option B: roll back to a specific commit (example hash)
  # git checkout a9554b5

  docker compose build
  docker compose up -d
  curl -s http://127.0.0.1:8765/api/config

  # Return to latest later:
  # git checkout main && git pull && docker compose up -d --build

ROLLBACK — desktop clone only

  cd ~/crm-kanban
  git fetch origin
  git checkout v1.0.0-production-2026-06-05   # or a commit hash

  # Return to latest:
  # git checkout main && git pull

Optional: keep last Docker image before risky upgrade (on server)

  docker tag onlyoffice-crm-kanban-dashboard:latest onlyoffice-crm-kanban-dashboard:backup-2026-06-05
  # Restore image later only if you did not rebuild from different code:
  # docker tag onlyoffice-crm-kanban-dashboard:backup-2026-06-05 onlyoffice-crm-kanban-dashboard:latest
  # docker compose up -d

Back up user profile data before major changes (server):
